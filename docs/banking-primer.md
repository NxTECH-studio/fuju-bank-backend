# 銀行システム入門 — ふじゅ〜中央銀行を作るための最小語彙

このドキュメントは、fuju-bank-backend の実装に入る前に「銀行システムとは何でできているのか」を 1 ページで掴むための学習用メモ。実装中は何度でもここに戻ってきてよい。

対象読者: Rails は書けるが、銀行・会計システムを作った経験はない人。

---

## 1. なぜ銀行は「ただの数値フィールド」ではダメなのか

素朴に作るとこうなる:

```ruby
class Artist
  # balance カラムに数値を入れて、入金時に +=、出金時に -= する
end
```

これが通用するのはおもちゃの段階までで、本物の銀行としては **ほぼすべての性質で破綻する**:

| 困ること | なぜ起きるか |
|---|---|
| バグで残高が合わなくなっても気付けない | 履歴がないので真の値が分からない |
| 「いつ・何が原因で」増えたか追えない | `balance` は結果しか持っていない |
| 同じリクエストを 2 回処理しても検知できない | 冪等性の仕組みがない |
| 並列書き込みで競合する | `UPDATE balance = balance + x` は race する |
| 監査・法規制に対応できない | 改ざん検知や履歴保全ができない |

銀行システムは「残高は **導出値 (derived)** であり、真実は **履歴 (log)** にある」という考え方で組み立てる。履歴さえ正しければ、残高はいつでも計算し直せる。逆に履歴が壊れていたら、それは銀行として致命的。

これが **台帳 (ledger) 中心設計** と呼ばれるもの。

## 2. 登場人物 — たった 4 つの概念

ふじゅ〜銀行に必要な銀行用語は、これだけ。

### 2.1 勘定科目 (Account)

「お金の置き場所」のカテゴリ。現実の銀行だと「普通預金」「当座預金」「貸付金」「現金」…と何百もあるが、**ふじゅ〜では 2 種類だけで始める**:

- **発行準備金 (`issuance_reserve`)** — 銀行が「これから世に放つ予定のふじゅ〜」を入れておく仮想の箱。1 つだけ存在する。
- **作家口座 (`artist`)** — 作家 1 人につき 1 つ。その作家が今まで受け取った分が貯まる。

> 💡 「口座」はユーザーそのものではなく、**ユーザーに紐付いた勘定科目**。`Artist has_one :account`。

### 2.2 仕訳 / ジャーナルエントリ (Journal Entry)

**1 つの取引**。銀行で何かが起きるたびに仕訳が 1 本生まれる。ふじゅ〜では「誰かが作品を 4.2 秒凝視したことで 15 ふじゅ〜が発行された」という出来事が 1 仕訳。

仕訳は単体では意味を持たない — 必ず **複数の明細行 (lines)** を従えていて、その明細の合計が 0 になったときにだけ完成する。

### 2.3 借方・貸方 (Debit / Credit) — ここが最大の山場

毎回の取引は **必ず 2 箇所以上のアカウントに同時に書き込まれる**。片方に「借方 (debit)」、もう片方に「貸方 (credit)」として。これが **複式簿記 (double-entry bookkeeping)** の核心。

ふじゅ〜の発行を例にすると:

```
取引: 鑑賞者の凝視により 15 ふじゅ〜を発行し、作家 42 に渡した

  借方 (debit)    作家口座 (artist_id=42)      15
  貸方 (credit)   発行準備金 (issuance_reserve) 15
                                               ---
                                                0  ← 必ずこうなる
```

読み方: 「作家 42 の口座が 15 増えた。その裏側で、発行準備金から 15 出ていった」。お金は無から湧いたのではなく、**銀行の別の場所から移動してきた** ことが構造的に表現されている。

どちらに書くかのルール（資産系 vs 負債系）は現実の簿記では暗記事項だが、**ふじゅ〜では「作家口座は借方で増える」「発行準備金は貸方で増える」 の 2 つだけ覚えれば十分**。あとは全部この 1 パターンを繰り返すだけ。

### 2.4 台帳 (Ledger)

すべての仕訳の明細行を時系列に並べたもの。「誰のアカウントに、いつ、いくら、借方/貸方として書き込まれたか」の **追記専用ログ (append-only log)**。

> ⚠️ **台帳の行は絶対に書き換えない・削除しない**。間違えたら「逆仕訳 (reversal)」を新しく追加して打ち消す。これも銀行の大原則。

## 3. 残高はどうやって出すか

残高 (balance) は **その場で計算するもの**、データベースに持つものではない:

```
作家42の残高 = SUM(作家42の借方) - SUM(作家42の貸方)
```

つまり台帳の該当行を集計するだけ。これが「履歴が真実、残高は導出値」の意味。

- 履歴が壊れない限り残高も壊れない
- 過去の任意の時点の残高も計算できる（`WHERE issued_at <= :time`）
- 残高を直接 UPDATE する必要がないので並列書き込みに強い

速度が問題になってきたら、別テーブルに集計値をキャッシュする（materialized projection / account_balances テーブル）という定石があるが、**ふじゅ〜では当面不要**。

---

## 4. 不変条件 (Invariants) — 銀行を銀行たらしめるもの

台帳システムが「壊れていない」ことは、次の 2 本の不等式で定義できる:

1. **仕訳内均衡**: どの仕訳 1 本を取っても、その中の借方合計 = 貸方合計
2. **系全体均衡**: データベース全体の借方合計 = 貸方合計

この 2 つさえ成り立っていれば、台帳は自己整合的である。これは **テストで強制できる** し、**DB 制約でも強制できる**（CHECK 制約やトリガー）。

fuju-bank-backend では:
- `JournalEntry` モデルの validation で (1) を守る
- spec に「全 spec 実行後に (2) が成り立つ」invariant test を 1 本置く

> 💡 この不変条件が壊れている状態は「銀行が壊れている」と同義。金融システムが堅牢に見える理由の 9 割はこの不変条件の存在。

## 5. 冪等性 (Idempotency) — 2 回押しても 1 回

マイニング層からのリクエストはネットワークを跨ぐので、**同じ encounter が 2 回届く可能性がある**（タイムアウト再送、クライアントバグ、etc）。素朴に作ると 2 回分発行されてしまう。

解決策: 各リクエストに **一意なキー (idempotency key)** を付けて、「このキーで既に仕訳を切ったか？」を DB で確認する。

ふじゅ〜では:

```
idempotency_key = "encounter:#{client_session_id}:#{observed_at}:#{artifact_id}"
```

`journal_entries.idempotency_key` に **unique index** を張っておく。2 回目の挿入は DB レベルで弾かれる → サービス層で既存レコードを返して 201 を返す。

> 💡 「DB 制約で守る」のがポイント。アプリ層だけで守ろうとすると、並列リクエストで race する。

## 6. ふじゅ〜固有の翻訳表

銀行用語 → ふじゅ〜世界の対応:

| 銀行用語 | ふじゅ〜での意味 |
|---|---|
| 勘定科目 | 発行準備金 / 作家口座 |
| 仕訳 (journal entry) | 1 回の発行 = 1 本 |
| 借方 (debit) | 作家口座が増える側 |
| 貸方 (credit) | 発行準備金が減る側 |
| 台帳 (ledger_lines) | 仕訳の明細行の集合 |
| 残高 | 作家が今持っているふじゅ〜の総量 |
| 通貨単位 | ふじゅ〜（整数で扱う。浮動小数は禁忌） |

**生データ vs 会計結果**: `gaze_events` は「視線の生の観測ログ」で、閾値未満の凝視もここに記録される（銀行が捨てないログ）。一方 `journal_entries` は「会計的にお金が動いた記録」で、閾値を超えたときだけ作られる。この 2 つは別テーブルで明確に分けておく。

## 7. API 契約書との関係

`docs/api-contract.md` の外形（レスポンスの JSON キー）は、この複式構造を **隠蔽** する。マイニング層や SNS 層からは「`ledger_entry_id` と `amount` が返ってきた」くらいにしか見えない。

内部のマッピング:

| API の概念 | 内部実装 |
|---|---|
| `ledger_entry_id` | `journal_entries.id` |
| `amount` | 該当仕訳の artist 側 ledger_line の `debit` 値 |
| `balance` | `SUM(debit) - SUM(credit)` on artist account |

外から見ると単なる "残高が 15 増えた" という話だが、内側では借方/貸方のペアが整合性を守っている。**API 契約書は一切変えない** まま、銀行らしい骨格を入れられる。
